# Addresses and signing

This page explains how Ligate Chain accounts are identified, how
transactions get signed, and **why MetaMask doesn't work today**.
Written for a reader who hasn't shipped a blockchain before.

## TL;DR

| Question | Answer today | Answer after [#72](https://github.com/ligate-io/ligate-chain/issues/72) |
|---|---|---|
| What format are Ligate addresses? | `lig1…` (Bech32m, 28 bytes) | Same. Plus `0x…` (20-byte EVM) becomes spendable too. |
| What signing scheme does the chain accept? | ed25519 only (via `RollupAuthenticator`) | ed25519 + secp256k1 (EVM) |
| Can MetaMask sign transactions for Ligate? | No | Yes |
| Can Solidity contracts deploy on the chain? | No (and not until v4, [#52](https://github.com/ligate-io/ligate-chain/issues/52)) | No (still v4) |

The rest of this doc unpacks each row.

## What an address actually is

An address on any blockchain is **like a bank account number**: a
public string that anyone can send tokens to, but only the holder of
the matching secret key can spend from.

Different blockchains use different formats:

| Blockchain | Format | Example |
|---|---|---|
| Bitcoin / Ethereum | Hex with `0x` prefix | `0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266` |
| Solana | Raw base58 | `83astBRguLMdt2h5U1Tpdq5tjFoJ6noeGwaY3mDLVcri` |
| Cosmos chains | Bech32 with chain prefix | `cosmos1xv9tklw7d82sezh9haa573wufgy59vmwe6xxe5` |
| **Ligate** | **Bech32 with `lig` prefix** | `lig1h72nh5c7jfjkcygku4thsh2t53dyh33kkpktpy84w06qwr4agvt` |

Ligate uses Bech32m encoding with the human-readable prefix `lig`,
configured in [`constants.toml`](../../constants.toml) as
`ADDRESS_PREFIX`. The `1` in `lig1…` is the Bech32 separator (between
the prefix and the data). It's not a version digit.

## What a "private key" actually is, and why it has to match

A private key is a **secret number**. Different blockchains use
different mathematical systems to convert that number into:

1. A public key (also a number, derived from the private one).
2. An account number (derived from the public key).
3. A signature scheme (a way to prove "I know the private key" without revealing it).

Ligate uses **ed25519** — the same math system Solana, Aptos, Sui,
and most modern non-EVM chains use. Ethereum and Bitcoin use
**secp256k1**.

These math systems are not interchangeable. **An ed25519 private key
cannot produce a valid secp256k1 signature, and vice versa.** They're
different cryptosystems.

Concretely:

- An Ethereum private key (e.g., one generated by MetaMask) signs
  transactions with secp256k1 + keccak256. It's mathematically
  married to a `0x…` address derived the Ethereum way.
- A Ligate private key signs with ed25519. It's mathematically
  married to a `lig1…` address derived the Sovereign-SDK way:
  `Bech32m("lig", first 28 bytes of SHA256(public_key))`.

You cannot use a Ligate key to sign for a `0x…` address, or an
Ethereum key to sign for a `lig1…` address. They live in separate
cryptographic universes.

## Why our chain rejects MetaMask transactions today

Our chain has exactly one transaction-authentication path right now:
[`RollupAuthenticator`](https://github.com/Sovereign-Labs/sovereign-sdk/blob/main/crates/module-system/sov-modules-api/src/runtime/capabilities/authentication.rs),
the SDK's standard ed25519 verifier. It takes incoming transaction
bytes, parses them, checks the ed25519 signature, derives the
sender's `lig1…` address from the signing public key, and dispatches
the call.

A MetaMask-signed transaction:

1. Is encoded as RLP (Ethereum's format), not borsh.
2. Is signed with secp256k1, not ed25519.
3. Authenticates a `0x…` address, not a `lig1…` one.

`RollupAuthenticator` understands none of those things. It rejects
the transaction at parse time. There is no second auth path that
*does* understand them, today.

This is why the `0x…` addresses we briefly had in `devnet/genesis/`
were a mistake: a `$LGT` balance sat there, but no signature
producible by anybody could ever spend it. The Anvil dev keys
(`0xac0974…`, `0x59c699…`, etc.) make secp256k1 signatures, which
our chain can't verify. Funds in those accounts were stuck forever.

We removed them in favour of `lig1…` addresses derived from real
ed25519 keys — see
[`crates/stf/tests/devnet_addresses.rs`](../../crates/stf/tests/devnet_addresses.rs)
for the deterministic derivation.

## What changes when [#72](https://github.com/ligate-io/ligate-chain/issues/72) lands

[Issue #72](https://github.com/ligate-io/ligate-chain/issues/72) adds
a **second auth path** to the chain. The runtime's `Auth` type
becomes a 2-variant enum (the SDK pattern):

```rust
enum LigateAuthenticator {
    Standard(ed25519_signed_tx),  // today's path, unchanged
    Evm(secp256k1_signed_rlp_tx), // new path
}
```

The `Evm` variant verifies a MetaMask-style RLP-encoded transaction:
it parses the RLP, verifies the secp256k1 signature, derives the
sender's `0x…` address, and dispatches the call to our **native
modules** (bank, attestation, etc.). A user signs with MetaMask, the
chain accepts it, and `$LGT` debits from their `0x…` account.

What this **does not** do: run Solidity contracts. The signed
transaction's payload is still a Ligate-native call (e.g.,
`AttestationCallMessage::SubmitAttestation`), not EVM bytecode.
There's no EVM execution environment until [#52](https://github.com/ligate-io/ligate-chain/issues/52) (v4).

So after #72:

- ✅ MetaMask, WalletConnect, viem, ethers, foundry can all sign Ligate transactions.
- ✅ Both `lig1…` and `0x…` addresses are spendable.
- ✅ Our `Auth` accepts two cryptographic universes side by side.
- ❌ You still can't deploy `MyContract.sol` on Ligate.
- ❌ ERC-20 tokens can't be created via Solidity. (`$LGT` and other tokens come via the curated `bank` module and the future `tokens` module — [#47](https://github.com/ligate-io/ligate-chain/issues/47)).

## What changes when [#52](https://github.com/ligate-io/ligate-chain/issues/52) lands (v4)

[Issue #52](https://github.com/ligate-io/ligate-chain/issues/52) adds
the SDK's `sov-evm` module to our runtime. That module is a real EVM
execution environment, so:

- Users can deploy Solidity contracts to Ligate Chain.
- Existing dApps (Uniswap clones, ERC-20 / ERC-721 implementations)
  deploy without rewrites.
- `$LGT` becomes a gas token in the EVM sense (a contract calls
  another contract, gas debits from the EVM account).

This is **explicitly deferred to v4**, after the attestation thesis
has earned the right to broaden surface area. It's a strategic
choice, not a missing feature.

## Mental model in one sentence

Today, **Ligate Chain is Solana-shaped**: ed25519 signing,
Bech32-encoded addresses, no EVM bytecode anywhere. The `0x…`
display capability that the address type carries is a UX hook for
a future #72 lane, not a current capability. Calling the chain
"EVM-compatible" today is misleading; calling it "EVM-friendly
on the v1 roadmap" is honest.

## Common confusions, addressed

**"Can I import my MetaMask seed phrase into Mneme?"**
Not directly. MetaMask seeds are BIP39 → secp256k1. Mneme's
keys are BIP39 → ed25519. The seed phrase format is the same, but
the derivation path lands at a different keypair. After #72, your
secp256k1 key still works for the `0x…` Lane, but the `lig1…` lane
needs an independently generated ed25519 key.

**"Then I have two accounts on the chain?"**
Yes, after #72. The same human can hold balance under their
`lig1…` (ed25519-derived) account and under their `0x…`
(secp256k1-derived) account, and they're cryptographically
distinct. Mneme can manage both, but they don't unify into one
balance. Unifying would require account abstraction, which is its
own design problem (not on the roadmap).

**"What about Privy?"**
Privy supports app-chains via embedded wallets. For Ligate, Privy
generates an ed25519 keypair inside the user's session and signs on
their behalf. Works today without #72. Privy's *MetaMask connector*
flow (where the user clicks "Connect MetaMask") is the path that
needs #72.

**"Doesn't the SDK use 28-byte hashed addresses to *enable* both
curves?"**
Yes. `Address(SHA256(pubkey)[..28])` lets the chain support any
public-key-shaped credential (ed25519, secp256k1, future curves)
through one address derivation. It's a useful design property even
when only one curve is wired today.
